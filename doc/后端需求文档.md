# 饱了么 (Baoleme) 智能双端系统：传统 SSM 后端开发及 API 对接设计规格书

> **重要架构调整说明**：由于后端舍弃了 Spring Boot 自动装配，改用**传统 SSM 框架（Spring 5.x + Spring MVC 5.x + MyBatis 3.x）**，本项目将基于传统的 **Maven 多模块/单模块骨架**，在常规 Servlet 容器（如 Apache Tomcat 9+）中运行。本文档已完全根据此传统架构进行了针对性优化，对 XML 配置文件、依赖声明、传统的项目树和双端（顾客端 Uni-app / 掌柜端 Uni-app）的实时连接对接进行了重构。

        <property name="url" value="${db.url}" />
        <property name="username" value="${db.username}" />
        <property name="password" value="${db.password}" />
        <property name="maxActive" value="20" />
        <property name="initialSize" value="5" />
    </bean>

    <!-- 3. 关联 MyBatis-Plus 自带的 SqlSessionFactory (替换传统原生MyBatis工厂，获得便捷方法能力) -->
    <bean id="sqlSessionFactory" class="com.baomidou.mybatisplus.extension.spring.MybatisSqlSessionFactoryBean">
        <property name="dataSource" ref="dataSource" />
        <property name="configLocation" value="classpath:mybatis-config.xml" />
        <!-- 扫描 XML 动态 SQL 的文件 -->
        <property name="mapperLocations" value="classpath:mapper/**/*.xml" />
    </bean>

    <!-- 4. Mapper 扫描器配置，加载底层 Mapper 接口 -->
    <bean class="org.mybatis.spring.mapper.MapperScannerConfigurer">
        <property name="basePackage" value="com.baoleme.mapper" />
        <property name="sqlSessionFactoryBeanName" value="sqlSessionFactory" />
    </bean>

    <!-- 5. 声明式事务切面管理 -->
    <bean id="transactionManager" class="org.springframework.jdbc.datasource.DataSourceTransactionManager">
        <property name="dataSource" ref="dataSource" />
    </bean>
    <tx:annotation-driven transaction-manager="transactionManager" />
</beans>

## 2. 数据库物理 DDL SQL (MySQL 一键导入)

在关系型数据库中建立以下表以支撑业务流及双端互动：

```sql
CREATE DATABASE IF NOT EXISTS `baoleme_db` CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci;
USE `baoleme_db`;

-- 1. 用户画像与偏好表 (对应 C端动态精准推荐)
CREATE TABLE `user_profile` (
  `id` VARCHAR(64) NOT NULL COMMENT '用户ID / 账号名',
  `name` VARCHAR(50) NOT NULL COMMENT '姓名/昵称',
  `tags` VARCHAR(255) DEFAULT '[]' COMMENT '标签JSON数组, 如 ["低卡轻食", "控糖", "偏爱辣味"]',
  `history_orders_count` INT DEFAULT 0 COMMENT '历史累计订单数',
  `favorite_category` VARCHAR(50) DEFAULT '热销推荐' COMMENT '最喜爱的分类名称',
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户画像偏好表';

-- 2. 菜品分类表
CREATE TABLE `category` (
  `id` VARCHAR(64) NOT NULL COMMENT '分类唯一标识',
  `name` VARCHAR(50) NOT NULL COMMENT '分类名称',
  `icon` VARCHAR(50) DEFAULT 'Utensils' COMMENT '前端显示图标类型',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_name` (`name`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='菜品分类表';

-- 3. 菜品表 (SKU库)
CREATE TABLE `dish` (
  `id` VARCHAR(64) NOT NULL COMMENT '菜品唯一标识',
  `name` VARCHAR(100) NOT NULL COMMENT '菜品名称',
  `price` DECIMAL(10, 2) NOT NULL COMMENT '单价',
  `description` TEXT COMMENT '菜品膳食配方、卡路里等营养描述',
  `image` VARCHAR(500) DEFAULT '' COMMENT '菜品主图Base64或URL路径',
  `category` VARCHAR(50) NOT NULL COMMENT '所属分类名称 (关联category.name)',
  `sales` INT DEFAULT 0 COMMENT '历史销量',
  `stock` INT DEFAULT 99 COMMENT '库存余量',
  `status` VARCHAR(20) DEFAULT 'active' COMMENT '上架状态: active(在售), inactive(下架)',
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='菜品SKU表';

-- 4. 订单主表
CREATE TABLE `orders` (
  `id` VARCHAR(64) NOT NULL COMMENT '订单单号',
  `user_id` VARCHAR(64) NOT NULL COMMENT '下单用户ID',
  `total_price` DECIMAL(10, 2) NOT NULL COMMENT '订单总支付价格',
  `status` VARCHAR(30) NOT NULL DEFAULT 'pending' COMMENT '状态: pending(待接单), accepted(已接/配餐中), cooking(制作中), delivering(配送中), completed(已送达), cancelled(已取消)',
  `created_at` VARCHAR(50) NOT NULL COMMENT 'ISO标准下单时间戳字符串',
  `address` VARCHAR(255) NOT NULL COMMENT '配送地址',
  `phone` VARCHAR(32) NOT NULL COMMENT '联系电话',
  `note` VARCHAR(255) DEFAULT '' COMMENT '订单备注/膳食偏好微调要求',
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='订单主表';

-- 5. 订单明细表 (购物车项级级联)
CREATE TABLE `order_item` (
  `id` BIGINT AUTO_INCREMENT COMMENT '主键',
  `order_id` VARCHAR(64) NOT NULL COMMENT '关联主订单ID',
  `dish_id` VARCHAR(64) NOT NULL COMMENT '菜品ID',
  `dish_name` VARCHAR(100) NOT NULL COMMENT '菜品快照名',
  `price` DECIMAL(10,2) NOT NULL COMMENT '价格快照',
  `quantity` INT NOT NULL COMMENT '购买数量',
  PRIMARY KEY (`id`),
  KEY `idx_order_id` (`order_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='订单菜品明细关联表';

-- 6. 客服会话工作流表 (双端并联人工客服分配)
CREATE TABLE `chat_session` (
  `id` VARCHAR(64) NOT NULL COMMENT '会话全局ID, 通常为 session_{userId}',
  `user_id` VARCHAR(64) NOT NULL COMMENT '关联用户ID',
  `user_email` VARCHAR(100) DEFAULT '' COMMENT '用户电子邮件邮箱地址',
  `mode` VARCHAR(20) DEFAULT 'bot' COMMENT '客服模式: bot(AI智能膳食客服), human(转人工客服)',
  `status` VARCHAR(20) DEFAULT 'active' COMMENT '状态: active(跟进中), resolved(已完结解决)',
  `last_updated` VARCHAR(50) NOT NULL COMMENT '最后更新时间戳',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_userid` (`user_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='客服呼叫分配会话表';

-- 7. 客服即时通信历史消息表 (包含AI大模型与人工的全部流聊天记录)

  `id` VARCHAR(64) NOT NULL COMMENT '消息单号',
  `session_id` VARCHAR(64) NOT NULL COMMENT '关联会话ID',
  `role` VARCHAR(20) NOT NULL COMMENT '发送方身份: user(顾客), assistant(AI助手), merchant(商家掌柜)',
  `content` TEXT NOT NULL COMMENT '消息文本',
  `timestamp` VARCHAR(50) NOT NULL COMMENT '发送时间戳',
  PRIMARY KEY (`id`),
  KEY `idx_session_id` (`session_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='即时通信交互明细表';
```

## 4. 双端即时人机/真人客服与 AI 推荐接口

饱了么系统核心业务亮点是 C 端的**智能营养师 + 智慧推荐算法**和 B 端掌柜一键协助连线的双端连通模式。

### 4.1 会话客服模式切换协议

由于是传统 SSM 项目，会话具有两个核心模式(`mode`)：
1.  `bot` (默认大模型智能辅导，此时由 Java 后端接收咨询后直联国内或官方大模型 API（如 Gemini）返回结果)。
2.  `human` (顾客点击“转人工”后，会话转接给处于在线状态的掌柜，其消息后续直接由 WebSocket 管道互通路由，不消耗大模型 Token)。

#### ① 获取当前会话实例
*   **Method**: `GET`
*   **Path**: `/api/chat/session`
*   **路径参数**: `?userId=xiaoming` (若不存在则后端默认根据 `INSERT` 产生一套默认会话)。

#### ② 客服模式流转变更
*   **Method**: `PUT`
*   **Path**: `/api/chat/session/{sessionId}/mode`
*   **请求参数 JSON**: `{ "mode": "human" }` —— 更新数据持久表。

#### ③ 获取最新聊天明细
*   **Method**: `GET`
*   **Path**: `/api/chat/messages/{sessionId}`

## 5. C2B 推荐算法实现 (AI 智能核心)

C 端“大模型智能精选/诊断推荐”由后端中转外部 API。推荐端点规格：

*   **Method**: `POST`
*   **Path**: `/api/gemini/recommend`
*   **输入实体 JSON**:
    ```json
    {
      "tags": ["低卡轻食", "控糖"],
      "craving": "我想吃一顿能抗饿的午餐，有肉和粗粮"
    }
    ```
*   **后端推荐整合器设计逻辑**:
    1.  从数据库查询出当前**在售且库存 > 0** 的菜品。
    2.  与用户的输入混合，拼接出推荐指令 Prompt：
        > "你现在是饱了么平台的智能营养顾问。现有库存如下：${dishes}。用户标签为 ${tags}，要求为：${craving}。请必须给出推荐理由和选出的最合适的 1-2 个菜品标识符（必须输出标准合法的 JSON 纯字符串，属性为：recommendationTitle, reason, items）"
    3.  调用 API 接口。
    4.  以 `application/json` 投递回前端呈现。

## 6. JSR-356 传统标准套接字通信 (Tomcat 级即时投屏)

因为不依托 Spring Boot Starter 开箱套件，建议在 Tomcat 环境中，直接使用标准 JSR-356 实现长连接，稳定性更佳且性能突出：

### 6.1 服务端编写范例
```java
package com.baoleme.websocket;

import javax.websocket.*;
import javax.websocket.server.PathParam;
import javax.websocket.server.ServerEndpoint;
import java.io.IOException;
import java.util.concurrent.ConcurrentHashMap;

@ServerEndpoint("/ws/baoleme/{role}/{userId}")
public class BaolemeWebSocketServer {
    // 采用高并发安全散列表存取全系统已连线的两端 Uni-app Channel 节点
    private static final ConcurrentHashMap<String, Session> clients = new ConcurrentHashMap<>();

    @OnOpen
    public void onOpen(Session session, @PathParam("role") String role, @PathParam("userId") String userId) {
        String key = role + "_" + userId;
        clients.put(key, session);
        System.out.println("双端就绪! 新增连接密钥: " + key);
    }

    @OnMessage
    public void onMessage(String message, Session session) {
        // 后端在这里处理普通的心跳包、消息包转发
        System.out.println("收到套接字帧: " + message);
    }

    @OnClose
    public void onClose(Session session, @PathParam("role") String role, @PathParam("userId") String userId) {
        String key = role + "_" + userId;
        clients.remove(key);
        System.out.println("拔出套接字: " + key);
    }

    /**
     * 单播向指定的目标客户端（如某个掌柜或顾客）精确推动语音报警或聊天消息
     */
    public static void sendToUser(String role, String userId, String jsonMsg) {
        String key = role + "_" + userId;
        Session target = clients.get(key);
        if (target != null && target.isOpen()) {
            try {
                target.getBasicRemote().sendText(jsonMsg);
            } catch (IOException e) {
                e.printStackTrace();
            }
        }
    }

    /**
     * 群发广播（可用来同时推送新订单语音给所有的在线掌柜）
     */
    public static void broadcastToRole(String rolePrefix, String jsonMsg) {
        clients.forEach((key, session) -> {
            if (key.startsWith(rolePrefix) && session.isOpen()) {
                try {
                    session.getBasicRemote().sendText(jsonMsg);
                } catch (IOException e) {
                    e.printStackTrace();
                }
            }
        });
    }
}
```

## 7. MyBatis-Plus 行级防超卖与防脏读重构
在传统 SSM 事务控制中，为避免答辩演示时遇到多端并发测试产生的数据脏读和透支：

```java
// 在 DishMapper.java 中使用 SQL 原生排他计算，避免超卖
public interface DishMapper extends BaseMapper<Dish> {
    @Update("UPDATE dish SET stock = stock - #{qty}, sales = sales + #{qty} " +
            "WHERE id = #{dishId} AND stock >= #{qty}")
    int updateStockAndSales(@Param("dishId") String dishId, @Param("qty") int qty);
}
```
通过 `stock >= #{qty}` 控制数据库行锁计算，由于返回受影响行数。若其返回值为 `0`，则代表库存不足。此时事务拦截自动触发 `@Transactional` 回滚机制，安全无缝！
